# Heap 영역

프로그램이 실행되는 동안 개발자가 동적으로 크기를 결정하고 할당할 수 있는 자유 메모리 공간

## 특징

- **동적 메모리 할당** : 런타임(실행될 때)에 크기가 결정되므로, 데이터의 크기를 예측할 수 없을 때 유용
- **개발자(또는 엔진)의 관리 필요** : 메모리를 할당했다면 반드시 해제해야 함
    - C# : 가비지 컬렉터(Garbage Collector)가 더이상 참조되지 않는 힙 모메리를 자동으로 청소
- **느린 접근 속도** : 메모리를 할당할 주소를 찾고, 포인터를 통해 접근해야 하므로 스택 영역에 비해 상대적으로 속도가 느림
- **포인터/참조를 통한 접근** : 힙에 저장된 데이터는 직접 접근할 수 없고, 그 데이터가 위차한 메모리 주소(참조값)를 통해서만 접근 가능

## 저장되는 내용

주로 크기가 가변적이거나 프로그램 전반에서 공유되어야 하고 수명이 긴 데이터들이 저장됨

- **런타임에 생성되는 객체(Object) 및 인스턴스(Instance)** : new 연산자로 생성하는 모든 클래스의 인스턴스
- **동적 배열 및 컬렉션** : 크기가 나변적인 동적 배열의 실제 데이터들
- **글로벌하게 참조된 데이터** : 특정함수가 끝나도 사라지지 않고, 여러 함수나 클래스에서 계속 참조해야 하는 데이터

## 스택 영역과의 상호작용

### 1. 참조 관계

스택에 저장된 변수가 힙 영역에 있는 실제 객체의 주소(포인터/참조)를 가리키는 방식으로 상호작용 발생

#### 예시(C#)

```csharp
void CreateMonster()
{

		Monster myMonster = new Monster();

}
```

- new Monster() (힙 영역) : 실제 몬스터의 체력, 마나 ,스탯 등 무거운 데이터는 힙 영역에 생성됨
- MyMonster (스태 영역) : 이 몬스터가 힙의 몇 번지 주소(예: 0x1234)에 있는지 그 주소값만 스택 영역의 지역변수에 저장됨
- 함수가 종료되면 스택에 있는 myMonster 변수는 사라지지만 힙에 있는 Monster 객체는 남아있음. 이후 가비지 컬렉터가 돌면서 참조 잃은 힙의 Monster 객체를 지움

### 2. 스택과 힙의 공간 싸움(충돌 가능성)

- 메모리 공간에서 스택은 높은 주소에서 낮은 주소 방향으로 쌓이고, 힙은 낮은 주소에서 높은 주소 방향으로 자라남.
- 두 영역은 같은 제한된 메모리 공간을 공유하기 때문에 힙 영역이 너무 커져서 스택 영역을 침번하면 Heap Overflow, 반대로 스택이 히힙을 침범하면 Stack Overflow가발생함

## 스택 영역과 비교

| 구분 | 스택(Stack) 영역 | 힙(Heap) 영역 |
| --- | --- | --- |
| 할당 시점 | 컴파일 타임(크기 고정) | 런타임(동적 할당, 크기 가변) |
| 저장 데이터 | 지역변수, 매개변수, 리턴값 | 객체(인스턴스), 동적배열데이터 |
| 해제 시점 | 함수 종료시 자동 해제(FIFO) | 개발자가 해제/GC가 수거할 때 |
| 접근 속도 | 매우 빠름 | 상대적으로 느림 |
| 자료 구조 | LIFO | 트리 구조 기반의 자유 공간 관리 |

## 주의

힙 영역은 동적인 데이터 처리를 가능하게 해주는 필수적인 공간이지만, 제대로 관리하지 않으면 메모리 누수나 가비지 컬렉션으로 인한 성능 저하(스투더링/렉)를 유발할 수 있어 효율적인 관리가 늘 중요한 영역
