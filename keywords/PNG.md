# PNG

| **항목** | **특징 및 명세** |
| --- | --- |
| **압축 방식** | 무손실 압축 (Lossless) |
| **핵심 알고리즘** | 예측 필터링(Paeth 등) + DEFLATE (LZ77 & Huffman) |
| **최대 색상 표현** | 48비트 트루컬러 (채널당 16비트 RGB) + 16비트 알파 채널 |
| **주요 장점** | 알파 채널(반투명) 완벽 지원, 텍스트/도형의 왜곡 없는 보존, 무결성 검사(CRC) 내장 |
| **주요 단점** | 고해상도 실사 사진의 경우 JPEG, WebP, AVIF 등에 비해 용량이 비대해짐 |

## 정의

- PNG(Portable Network Graphics)
- 비손실 그래픽 파일 포맷의 표준
- GIF 포맷의 특허 문제(LZW 알고리즘의 라이선스 파동)를 해결하고 인터넷 환경에 최적화된 포맷을 제공하기 위해 1996년 W3C에서 표준화
    - GIF와의 차이점 **:** GIF의 256색 제한을 극복하여 트루 컬러(True Color)를 지원하며, 더 정교한 알파 채널(투명도)을 제공
    - JPEG과의 차이점 **:** JPEG은 손실 압축(Lossy) 방식이라 사진에 유리하지만, PNG는 **비손실 압축(Lossless)** 방식이므로 텍스트, 라인 아트, 아이콘처럼 경계가 명확한 이미지에 유리함.

## 특징

- **무손실 압축 (Lossless Compression) :** 이미지 데이터를 압축할 때 화질 저하(손실)가 전혀 발생 X. 여러 번 저장해도 원본 화질을 그대로 유지.
- **알파 채널(Alpha Channel) 지원 :** 단순 투명뿐만 아니라 반투명(0~255 단계) 효과를 완벽하게 지원. 덕분에 배경이 있는 웹페이지나 그래픽 합성 작업에 필수적.
- **다양한 색상 심도 (Color Depth) :**
    - **PNG-8 :** 최대 256색만 표현 (GIF 대체용, 용량이 작음)
    - **PNG-24 :** 약 1,600만 색상의 트루 컬러 지원 (RGB 각 8비트)
    - **PNG-32 :** 트루 컬러(24비트)에 알파 채널(8비트)이 추가된 형태
- **인터레이싱 (Interlacing) :** 이미지가 다운로드 될 때 위에서부터 느리게 열리는 대신, 처음에는 모자이크처럼 흐릿하게 보이다가 점점 선명해지는 기술(Adam7 알고리즘)을 지원.

## PNG 파일의 내부 구조

### ① 파일 시그니처 (8 바이트)

- 모든 PNG 파일은 항상 동일한 8바이트 데이터로 시작 → 프로그램이 이 파일을 읽을 때 "이것은 정상적인 PNG 파일이다"라고 인식하는 기준이 됨.
    - **16진수 표기:** `89 50 4E 47 0D 0A 1A 0A`
    - **텍스트 표기:** `\x89 PNG \r \n \x1A \n`

### ② 청크 (Chunk) 구조

- 시그니처 이후의 모든 데이터는 '청크'라는 단위로 쪼개져 저장됨.
- 각 청크는 다음과 같은 고정된 형식을 가집니다.

| **영역 (Field)** | **크기** | **설명** |
| --- | --- | --- |
| **Length** | 4 바이트 | Chunk Data의 크기를 나타냄 |
| **Chunk Type** | 4 바이트 | 청크의 종류를 나타내는 4글자 아스키(ASCII) 코드 |
| **Chunk Data** | 가변적 | 청크 유형에 따른 실제 데이터 영역 |
| **CRC** | 4 바이트 | 데이터 오류 검출을 위한 순환 중복 검사 코드 |

### ③ 핵심 필수 청크 (Critical Chunks)

PNG 파일이 정상적으로 열리려면 **반드시** 아래 4가지 청크가 순서대로 존재해야 함. (유형 이름의 첫 글자가 대문자임)

1. **IHDR (Image Header):** 가장 먼저 나오는 청크로, 이미지의 가로/세로 크기, 색상 심도, 압축 방법, 필터 방법, 인터레이스 방식 등의 기본 메타데이터가 담겨 있음.
2. **PLTE (Palette):** 인덱스 색상(PNG-8 등)을 사용할 때만 필수적인 청크로, 사용된 색상들의 인덱스 표가 들어있음. 트루 컬러 이미지에서는 생략될 수 있음.
3. **IDAT (Image Data):** 실제 이미지 픽셀 데이터가 저장되는 곳. 용량이 크기 때문에 하나의 파일에 여러 개의 IDAT 청크로 나누어 저장되기도 함.
4. **IEND (Image End):** PNG 파일의 끝을 알리는 청크로, `Chunk Data`는 비어있음.

### ④ 보조 청크 (Ancillary Chunks)

필수는 아니지만 이미지의 부가 정보를 담는 청크(첫 글자가 소문자임)

- `tRNS` (Transparency) : 팔레트나 특정 색상을 투명하게 처리할 때 사용
- `gAMA` (Gamma) : 모니터 환경에 따른 감마 보정 값
- `tEXt` / `zTXt`: 텍스트 메모, 저작권 정보, 촬영 기기 등 메타데이터 저장

## PNG의 압축 알고리즘

1. **필터링 (Filtering - 예측 부호화) :**
이미지 픽셀은 주변 픽셀과 색상이 비슷할 확률이 높음. PNG는 현재 픽셀 값 대신, 왼쪽/위쪽 픽셀을 바탕으로 현재 픽셀 값을 '예측'한 뒤 그 차이 값(결과값)만 기록. 차이 값은 0에 가까운 숫자가 많아지므로 압축 효율이 극대화. (Sub, Up, Average, Paeth 필터 등이 사용됨)
2. **디플레이트 (Deflate) :**
필터링을 거친 데이터를 **LZ77** 알고리즘과 허프만 코딩(Huffman Coding)을 결합한 무손실 압축 방식인 Deflate(ZIP 파일에서 쓰는 방식)로 최종 압축함.

## PNG의 장점과 단점

### 장점

- 글자, 로고, 라인 아트, 스크린샷처럼 경계가 명확하고 색상 반전이 심한 그래픽을 화질 저하 없이 깨끗하게 저장.
- 배경을 투명하게 만들 수 있어 웹 디자인 및 UI/UX 디자인에 필수적.
- 특허료가 없는 오픈 표준이라 어떤 프로그램에서든 제약 없이 사용 가능.

### 단점

- **큰 파일 용량 :** 사진처럼 색상이 복잡하고 그라데이션이 많은 이미지는 JPEG(손실 압축)에 비해 용량이 지나치게 커짐.
- **애니메이션 미지원 :** 기본 PNG는 움직이는 이미지를 지원하지 않음. (이를 해결하기 위해 APNG 포맷이 나왔으나 표준화가 늦어 GIF에 비해 범용성이 떨어졌으나 현재는 대부분의 브라우저가 APNG를 지원함.)
- **CMYK 미지원 :** 인쇄용 색상 영역인 CMYK를 지원하지 않고 RGB 기반의 디지털 디스플레이 전용으로 설계됨.

## 사용 시기

- **PNG를 써야 할 때:** 로고, 아이콘, 투명 배경이 필요한 이미지, 텍스트가 포함된 그래픽, 화질 손상이 절대 없어야 하는 원본 편집용 이미지
- **피해야 할 때:** 고해상도 디지털카메라 사진 (용량이 너무 커지므로 JPEG나 WebP 권장), 인쇄소에 보낼 출력용 작업물
