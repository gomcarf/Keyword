# TCB - Thread Status

- 운영체제(OS)가 멀티스레딩 환경에서 스레드의 효율적인 스케줄링과 자원 관리를 위해 정의한 스레드의 생명주기(Lifecycle) 단계를 의미
- OS의 스케줄러는 각 스레드의 상태 정보를 담고 있는 TCB(Thread Control Block, 스레드 제어 블록)를 실시간으로 확인하며, 어떤 스레드에 CPU를 할당하고 어떤 스레드를 대기시킬지 결정

## 정의

- 프로세스 내에서 실행되는 독립적인 흐름(스레드)이 현재 어떤 실행 조건에 처해 있는지를 나타내는 메타데이터
- OS는 시스템 자원(CPU, 메모리, I/O 장치 등)의 낭비를 막고 CPU 효율을 극대화하기 위해 스레드를 여러 상태로 나누어 관리

## 주요 기능 및 특징

1. **문맥 교환의 핵심 지표** : 스레드의 상태가 `Running ──> Ready` 혹은 `Running ──> Blocked`로 바뀔 때, OS는 현재 CPU 레지스터의 중간 계산 값을 해당 스레드의 TCB에 저장하고, 새로 실행될 스레드의 TCB 정보를 CPU에 복원. 스레드 상태 값은 이 무거운 작업(콘텍스트 스위칭)을 언제 유발해야 하는지 판단하는 기준이 됨.
2. **CPU 스케줄러와의 상호 작용** : OS 스케줄러는 수많은 스레드 중 오직 Ready 상태인 스레드만을 대상으로 다음 실행 대상을 선정. 만약 상태 관리가 제대로 되지 않아 Blocked 상태의 스레드에 CPU를 주게 된다면, CPU 성능의 막대한 낭비(Starvation, Idle 상태 지속 등)가 발생
3. **동기화 및 데드락(Deadlock) 감지** : 여러 스레드가 동일한 자원(예: DB 연결, 전역 변수)을 쓸 때, 먼저 도착한 스레드가 자원을 선점하면 뒤늦은 스레드들은 `Blocked` 상태로 전환. OS와 개발자는 스레드의 상태를 모니터링하여 어떤 스레드가 락(Lock)을 쥐고 안 놓아주는지, 혹은 서로 자원을 기다리며 무한 대기하는 데드락(Deadlock)에 빠졌는지 감지할 수 있음.

## 스레드 상태

```csharp
[ New ] ──> [ Ready ] <──(Time Slice Exceeded)──> [ Running ] ──> [ Terminated ]
				▲                                     │
				└─────── [ Blocked / Waiting ] <──────┘ (I/O, Lock, Sleep)
```

### ① New (생성 / 초기화)

- **설명 :** 스레드 객체가 메모리에 생성되었으나, 아직 OS 스케줄러의 Ready Queue에 등록되지 않아 실행될 수 없는 전 단계.
- **특징 :** TCB 공간이 메모리에 할당되고 스레드 ID, 초기 우선순위 등 기본 정보만 세팅된 상태.

### ② Ready (준비)

- **설명 :** 실행에 필요한 모든 자원(메모리 등)을 확보한 채, 오직 CPU 할당(선택)만을 기다리고 있는 상태.
- **특징 :** Ready Queue에서 대기하며, OS 스케줄러가 알고리즘에 따라 이 상태의 스레드 중 하나를 선택(Dispatch)해 실행 상태로 바꿈.

### ③ Running (실행)

- **설명 :** 스레드가 실제로 CPU를 점유하고 프로세서의 명령어를 실행 중인 상태.
- **특징 :** CPU의 PC(Program Counter)나 SP(Stack Pointer) 레지스터가 이 스레드의 정보를 가리킵니다. 멀티코어 환경이라면 여러 스레드가 동시에 Running 상태를 가질 수 있음.

### ④ Blocked / Waiting (보류 / 대기 / 블록)

- **설명 :** 스레드가 작업을 계속하기 위해 특정 이벤트나 자원의 해제를 기다리는 상태.
- **특징 :** CPU를 받아봤자 일을 할 수 없는 상태(예: 파일 읽기 완료 대기, 네트워크 응답 대기, 동기화 Lock 획득 대기, `sleep()` 호출)이므로 CPU 소모를 막기 위해 Ready Queue에서 제외되어 Blocked Queue로 이동. 대기 조건이 만족되면 다시 Ready 상태로 돌아감.

### ⑤ Suspended (지연 / 중지)

- **설명 :** 시스템의 메모리가 극도로 부족할 때, OS가 메모리 확보를 위해 스레드(또는 부모 프로세스)를 메모리에서 디스크(Swap 영역)로 일시적으로 쫓아낸(Swap-out) 상태.
- **특징 :** 메모리에서 내려왔기 때문에 실행이 불가능하며, 메모리가 확보되거나 원인 이벤트가 해결되어 메모리로 다시 로드(Swap-in)될 때까지 대기. (Suspended Ready, Suspended Blocked 등으로 세분화)

### ⑥ Terminated (종료 / 소멸)

- **설명 :** 스레드가 자신의 실행 코드를 모두 마쳤거나, 에러로 인해 강제 종료(Aborted)된 상태.
- **특징 :** 스레드가 할당받았던 스택 메모리와 자원은 해제되지만, OS나 부모 프로세스가 종료 코드(Exit Code)를 확인할 때까지 TCB 정보는 일시적으로 메모리에 남을 수 있음(좀비 상태). 확인이 끝나면 TCB까지 완전히 삭제.

## PCB의 Process Status와의 차이

| **비교 항목** | **PCB - Process Status (프로세스 상태)** | **TCB - Thread Status (스레드 상태)** |
| --- | --- | --- |
| **개념적 의미** | 자원 소유 및 실행 환경의 거시적 상태 | 실제 CPU 연산 및 실행 흐름의 미시적 상태 |
| **관리 단위 (Scope)** | 프로그램(프로세스) 전체 단위 | 프로세스 내부의 개별 실행 흐름(스레드) 단위 |
| **CPU 점유 주체** | 프로세스는 직접 CPU를 점유하지 않음<br>*(내부 스레드가 점유함을 보장하는 단위)* | 실제 CPU 코어를 직접 점유하여 명령어를 실행함 |
| **독립성 (Isolation)** | 프로세스가 다른 프로세스 상태에 영향을 주지 않음 | 같은 프로세스 내의 스레드끼리 서로 상태 전이에 영향을 줄 수 있음 *(예: Lock 대기)* |
| **상태의 상호 영향** | 프로세스가 `Blocked`되면 내부의 모든 스레드가 동작을 멈춤 | 스레드 A가 `Blocked`되어도, 스레드 B는 여전히 `Running` 가능 |
| **주요 전이 원인** | * 프로그램의 시작 및 완전히 종료<br>* 대규모 시스템 자원(파일, 소켓 등) 할당 및 해제<br>* 메모리 부족으로 인한 Swap-out | * 스케줄러의 타임 슬라이스 만료 (Timeout)<br>* 스레드 간 동기화 락(Lock) 획득 실패<br>* 코드 내 `sleep()`, `yield()` 등의 호출 |
| **소멸 (Terminated)** | 프로세스가 가진 모든 메모리 영역(Code, Data, Heap)과 파일 자원이 완전히 OS에 반환됨 | 스택 영역만 정리되고, TCB는 부모 프로세스나 OS가 결과를 확인하기 전까지 일시적으로 남음 |
